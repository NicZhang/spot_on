# Spot On（约球） — 产品需求文档（PRD）

> **文档版本：** v1.0
> **更新日期：** 2026-02-15
> **产品形态：** 微信小程序
> **文档状态：** 初稿

---

## 一、产品概述

### 1.1 产品定位

Spot On（约球）是一款面向中国业余足球圈的**球队约战与管理工具**，以微信小程序为载体，解决业余球队在"找对手、管报名、收球费、记战绩"四个环节中的核心痛点。

### 1.2 产品愿景

让每一位业余球队队长从"接龙统计员、人肉计算器、催款讨债鬼"的角色中解放出来，让每一位球员拥有属于自己的"足球生涯档案"。

### 1.3 目标市场

- **地域：** 中国一二线城市（球场资源密集、业余足球活跃）
- **赛制：** 以 5v5、7v7、8v8 为主的非正式比赛（非联赛/杯赛）
- **用户规模预估：** 中国业余足球注册球队约 10 万+，活跃球员超 150 万

### 1.4 产品边界（微信小程序约束）

微信小程序作为载体，存在以下边界，产品设计必须在此范围内：

| 约束项 | 说明 | 产品影响 |
|--------|------|----------|
| 主包体积 ≤ 2MB | 分包后单包 ≤ 2MB，总包 ≤ 20MB | 功能按模块分包加载 |
| 消息推送受限 | 仅支持订阅消息（用户主动授权，一次授权一次推送） | 无法实现传统 Push 通知，关键节点引导用户授权 |
| 支付能力 | 仅支持微信支付，不支持资金池/代扣 | 不做"钱包充值"，每次交易独立支付 |
| 后台运行受限 | 小程序进入后台约 5 分钟后挂起 | 不做实时功能（如实时比分直播） |
| 定位精度 | 微信定位 API 精度约 30-100m | 签到打卡只做参考，不作为强制仲裁依据 |
| 无长连接 | 不支持原生 WebSocket 长连接场景 | 不做实时聊天，引导回微信群沟通 |

### 1.5 关键产品决策（调研需求取舍说明）

本产品需求基于对暴风联队队长"老雷"的深度访谈，但**并非所有用户诉求都应直接转化为产品功能**。以下是经过产品判断后的核心取舍逻辑：

#### A. 采纳并纳入 V1 的需求

| 调研原始需求 | 纳入理由 | 落地形式 |
|-------------|---------|---------|
| 竞技队与养生队分开匹配 | 用户最强烈的需求，且实现成本低（一个筛选字段） | 球队创建时自选"竞技型/养生型"，约战大厅按池子分列 |
| 球衣颜色互斥 | 高频痛点（几乎每次约战都会遇到），开发成本极低 | 约战发布时选颜色，应战时系统校验色系冲突 |
| 赛后 AA 账单 + 系统催收 | 队长痛点最深的环节（"像讨债的"），且是产品替代微信群的核心差异化 | 队长录入费用→系统算账→推送账单→超时自动提醒 |
| 比分双向确认 | 直接影响战绩数据的准确性，是信用体系的基石 | 主队提交→客队确认/异议→超时默认机制 |
| 进球/助攻"关门自治" | 用户原话："对手不关心你们队谁进的球"。将跨队确认和队内分配解耦，大幅降低争议概率 | 比分确认是跨队行为，进球分配是队长的队内权限 |
| 球队信用分 | 解决"鸽子队"和"赖皮队"问题的根基。用户愿意为"靠谱"付出筛选成本 | 履约加分+爽约扣分，展示层降权（非封号） |
| 球员个人数据档案 | 驱动普通球员使用小程序的核心诱饵——"社交货币"。没有球员参与，产品闭环跑不通 | 累计出场、进球、助攻、MVP 次数 |
| 球队性别构成标注 | 用户强调"不是歧视，是知情权"。不标注会导致到场后的尴尬和安全隐患 | 创建球队时必填，约战时可筛选 |
| MVP 投票 | 实现成本低，但对球员留存的激励效果极强（"为了 MVP 金星标志，大家都会来"） | 赛后 24 小时内队内匿名投票 |

#### B. 调研中强烈要求但不纳入（或降级处理）的需求

| 调研原始需求 | 不纳入/降级原因 | 替代方案 |
|-------------|----------------|---------|
| **Elo 天梯积分系统** | 业余足球人员流动大（每场阵容不同），数据采集依赖队长手动录入（准确性存疑）。冷启动阶段样本不足，评分反而会引发更多争议（"凭什么我才 1100 分？"） | V1 用"自主定位（竞技/养生）+ 年龄段 + 互评标签"组合替代。V2 数据积累充足后再考虑引入 |
| **球队钱包 / 预充值自动扣费** | 用户最想要的功能（"没有之一"），但**在中国，小程序运营方持有用户预存资金属于"资金池"，需持有《支付业务许可证》**。无牌照运营资金池属违法行为，监管风险不可接受 | V1 采用"赛后即时微信支付"，用系统催收替代人工催款。未来可考虑与持牌机构合作 |
| **约战押金 / 违约金冻结** | 微信支付不支持"预授权冻结"能力（需企业资质+专项审批），且押金纠纷本身会成为新的客诉来源 | V1 用信用分机制软约束（低信用分球队在列表中降权/置底）。V2 评估接入第三方担保支付 |
| **GPS + 照片强制签到** | 微信定位 API 精度仅 30-100m，城市球场往往密集分布（相邻球场可能仅 50m），误判率高。且强制拍照打卡打断比赛前的自然体验 | V1 做"队长手动确认对方到场"（主队确认客队），不做技术强制。签到数据仅作参考 |
| **双盲互评机制** | 前端需维护"双方都提交后才可见"的状态机，后端需要额外的可见性控制逻辑，V1 开发成本较高 | V1 做单向标签评价 + 聚合展示（同一标签 ≥3 个不同对手标记才公开）。V2 引入双盲 |
| **裁判预约平台** | 供给侧（裁判资源）冷启动极难——需要地推招募裁判、建立定价体系、处理排期调度和纠纷，本质上是一个独立的双边市场 | V1 不做。V2 以"信息撮合"模式试水（裁判自注册，平台不担保） |
| **赛后战报海报生成** | 需要设计多套模板、服务端渲染图片、处理字体/排版适配，开发量不小且非核心闭环功能 | V1 仅做数据展示（比分+进球+MVP 在详情页呈现）。V2 引入海报模板+一键分享 |

#### C. 调研中提出但判定为伪需求或超出产品边界的需求

| 调研原始需求 | 判定理由 |
|-------------|---------|
| **给对手看的战术阵型图** | 用户自己承认"阵型开场 5 分钟就散了"。业余 7v7 默认 3-2-1，无战术分析价值。真实需求是"上场时间公平分配"，而非战术板 |
| **请边裁** | 7v7/8v8 小场地无严格越位规则，用户原话："有那个预算我宁愿买箱运动饮料"。需求频次趋近于零 |
| **请啦啦队** | 与目标用户（30-45 岁中年男性、大排档赛后文化）的真实场景严重脱节。用户原话："我要是敢跟队员说请了啦啦队，信不信第二天我的腿就被媳妇打断" |
| **AI 智慧球场 / 自动视频剪辑** | 需要硬件基础设施投入（球场摄像头+边缘计算设备），属于独立的 ToB 业务线，不是微信小程序能承载的能力 |
| **实时比分直播** | 小程序后台运行 5 分钟后挂起，无法维持长连接。且业余比赛没有"边踢边记分"的操作习惯——都是踢完再回忆 |
| **球队定制酒水——平台直营电商模式** | 若 Spot On 自己下场卖酒，需持有食品经营许可证（含酒类）、仓储物流等重资产投入，不适合轻量小程序。但可通过"场景导流 + 外部交易"模式规避（详见模块十） |
| **保险嵌入销售** | 互联网保险销售需持有或合作保险经纪牌照（银保监会监管）。合规准备周期长，V3 阶段提前布局 |
| **"女球员进球算 2 分"等特殊规则** | 过于细碎的规则协商，系统化收益低、开发成本高。备注字段已可满足"君子协定"的沟通需求 |

#### D. 决策原则总结

1. **合规红线不碰：** 资金池、保险销售等需要牌照的能力，宁可功能缺失也不触碰监管底线
2. **平台边界清晰：** 小程序是"轻工具"，不是"重平台"。视频、电商、保险等重业务不在 V1 范围内
3. **数据驱动迭代：** Elo 评分、双盲互评等需要大量数据支撑的功能，等用户密度到位后再引入
4. **解决 80% 的痛而非 100% 的痒：** 队长的核心痛（催款尴尬、报名混乱、找不到靠谱对手）优先于球员的锦上添花（战报海报、视频集锦）
5. **尊重用户判断，但不盲从：** 用户提出的需求反映真实痛点，但用户构想的解决方案不一定是最优路径（如用信用分降权替代押金冻结）

## 二、目标用户

### 2.1 用户画像

| 维度 | 描述 |
|------|------|
| 性别 | 男性为主（>95%） |
| 年龄 | 25-45 岁，核心群体 30-40 岁 |
| 职业 | 公司职员、中小企业主、自由职业 |
| 踢球频率 | 每周 1-2 次 |
| 赛制偏好 | 7v7 / 8v8 为主，少量 5v5 / 11v11 |
| 组织方式 | 微信群接龙 + 群收款 |
| 核心诉求 | 找靠谱对手、省心管队、记录回忆 |

### 2.2 用户角色

| 角色 | 定义 | 核心诉求 | 使用频率 |
|------|------|----------|----------|
| **队长（Captain）** | 球队创建者/管理员，负责约战、报名管理、收费、赛后记录 | 效率工具：减少重复劳动、避免催款尴尬 | 高频（每周 2-3 次） |
| **球员（Player）** | 球队成员，参与报名、支付、投票 | 个人数据档案、公平透明、社交炫耀 | 中频（每周 1-2 次） |
| **散客（Free Agent）** | 未加入固定球队，寻找临时参赛机会 | 快速找球踢、无社交负担 | 低频（按需） |

> **V1 优先级：** 队长 > 球员 > 散客。散客功能列入 V2 规划。

### 3.1 模块一：用户与认证

#### 3.1.1 微信登录

- **触发时机：** 首次使用任何需要身份的功能时弹出授权引导（非强制拦截首页）
- **流程：**
  1. 调用 `wx.login()` 获取临时 code
  2. 后端用 code 换取 openid，创建/匹配用户记录，返回 JWT Token
  3. 前端缓存 Token，后续请求携带
- **获取信息：** 微信头像、昵称（需用户主动点击头像/昵称组件授权，微信最新规范）
- **补充资料（可选）：** 常用位置、擅长位置（前锋/中场/后卫/门将）、年龄段

#### 3.1.2 用户身份

- 一个微信用户 = 一个平台账号
- 一个账号可以加入多支球队（V1 限制最多 3 支）
- 一个账号可以创建最多 1 支球队（队长身份）

### 3.2 模块二：球队管理

#### 3.2.1 创建球队

队长填写以下信息创建球队：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 球队名称 | string | ✅ | 2-16 字符，不可重名 |
| 球队 Logo | image | ❌ | 支持上传或选择预设模板 |
| 所在城市 | string | ✅ | 省-市二级选择 |
| 常用球场 | string | ❌ | 手动输入或地图选点 |
| 赛制偏好 | enum | ✅ | 5v5 / 7v7 / 8v8 / 11v11（多选） |
| 球队性别 | enum | ✅ | 纯男足 / 纯女足 / 男女混合 |
| 球队定位 | enum | ✅ | 竞技型 / 养生型（影响匹配池） |
| 平均年龄段 | enum | ✅ | 20-25 / 25-30 / 30-35 / 35-45 / 45+ |
| 球队简介 | string | ❌ | 最多 200 字 |

**产品决策说明：**
- **"竞技型"与"养生型"分池**：调研中用户强烈要求区分"想赢"和"图乐"的球队。V1 通过自主申报实现（非算法强制），两个池子的约战列表独立展示。
- **不做 Elo 天梯积分**：调研中用户提出了完整的 Elo 评分构想，但业余足球人员流动大、数据采集成本高，Elo 系统冷启动困难。V1 用"自主定位 + 互评标签"替代，V2 根据数据积累再考虑引入。

#### 3.2.2 球队成员管理

- **邀请方式：** 队长生成小程序码/分享卡片，球员扫码申请加入
- **审批机制：** 队长审批通过后成为正式成员
- **角色权限：**

| 角色 | 权限 |
|------|------|
| 队长 | 全部管理权限（约战、报名、结算、数据录入、成员管理） |
| 副队长 | 协助管理（报名管理、数据录入），最多 2 人 |
| 队员 | 报名、支付、投票、查看 |

- **移除成员：** 队长可移除队员（需二次确认），被移除者保留历史数据但无法查看球队内容
- **退出球队：** 球员可主动退出，退出后无法查看球队内部信息

#### 3.2.3 球队主页（公开）

外部用户（约战对手）可见的球队信息：

- 球队名称、Logo、城市、赛制偏好
- 球队定位（竞技/养生）、平均年龄段、性别构成
- 成立时间、总比赛场次、近 10 场战绩（胜/平/负）
- 信用分（见 3.7）
- 互评标签云（见 3.7.2）
- 是否有专职门将（调研中被认为是判断球队实力的重要指标）
- 是否有统一队服

### 3.3 模块三：约战匹配

这是产品的核心模块，解决"找谁踢"的问题。

#### 3.3.1 发布约战（"我有场，找对手"）

队长发布一条约战信息，包含：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 比赛日期 | date | ✅ | 仅可选未来 14 天内 |
| 开球时间 | time | ✅ | — |
| 比赛时长 | enum | ✅ | 60min / 90min / 120min |
| 赛制 | enum | ✅ | 5v5 / 7v7 / 8v8 / 11v11 |
| 球场名称 | string | ✅ | 手动输入或从历史记录选择 |
| 球场位置 | location | ✅ | 地图选点（用于距离计算） |
| 场地费总额 | number | ✅ | 单位：元 |
| 费用分摊方式 | enum | ✅ | 各付各的 / AA平摊 / 主队全包 |
| 本方球衣颜色 | color picker | ✅ | 预设 12 色可选 |
| 预计到场人数 | number | ✅ | 例如 10-12 人 |
| 约战池 | enum | ✅ | 竞技池 / 养生池（默认跟随球队定位） |
| 性别要求 | enum | ✅ | 不限 / 仅匹配纯男足 / 仅匹配纯女足 |
| 备注 | string | ❌ | 自由填写（如"需要自备饮水"等） |

**产品决策说明：**
- **球衣颜色互斥**：发布方选定颜色后，接单方在应战时需选择己方颜色，系统校验色系是否冲突（同一色系提示更换）。这是调研中反馈的高频痛点。
- **不做"有队找场"模式（V1）**：调研中用户提出"我没场地，找球赛"的需求。但此模式涉及散客匹配和场地资源整合，复杂度高，V1 仅支持"有场找对手"的标准模式，V2 考虑"加入公开赛"功能。

#### 3.3.2 约战大厅（浏览与筛选）

所有已发布且未被接单的约战信息，以列表形式展示。

**筛选条件：**

| 筛选项 | 类型 | 说明 |
|--------|------|------|
| 距离 | range | 默认 10km 内，可选 5/10/20/不限 |
| 赛制 | enum | 5v5 / 7v7 / 8v8 / 11v11 |
| 日期 | date range | 今天 / 明天 / 本周 / 自选 |
| 约战池 | enum | 竞技 / 养生 |
| 对手年龄段 | enum | 20-25 / 25-30 / 30-35 / 35-45 / 45+ |
| 性别 | enum | 不限 / 纯男足 / 纯女足 |

**列表卡片信息：**
- 发起球队名称 + Logo
- 日期、时间、赛制
- 球场名称 + 距离（km）
- 费用分摊方式 + 金额
- 球队信用分 + 年龄段标签
- 球衣颜色色块

#### 3.3.3 应战流程

1. 客队队长点击"应战"，选择己方球衣颜色（系统校验颜色冲突）
2. 填写己方预计到场人数
3. 提交应战申请
4. **主队队长审批**：查看客队球队主页（战绩、信用分、互评标签），决定接受或拒绝
5. 接受后，双方进入"已确认比赛"状态

**产品决策说明：**
- **不做自动匹配**：调研表明队长非常在意对手的选择权（"我要看看对方是不是靠谱"），所以采用"发布-浏览-申请-审批"的人工模式，而非系统自动匹配。
- **V1 不做押金/保证金**：调研中用户强烈要求"违约金机制"。但微信小程序无法实现资金冻结（需持有支付牌照），V1 仅用信用分机制约束履约，V2 考虑接入第三方担保支付能力。

#### 3.3.4 约战状态机

```
[待接单] → 有人应战 → [待审批]
[待审批] → 主队接受 → [已确认] → 比赛结束 → [待结算]
[待审批] → 主队拒绝 → [待接单]
[已确认] → 任一方取消 → [已取消]（触发信用分扣减规则）
[待接单] → 超过比赛时间无人接单 → [已过期]
```

### 3.4 模块四：报名与出勤

约战确认后，双方各自在队内发起报名。

#### 3.4.1 发起报名（队长操作）

- 队长点击已确认的比赛 → "发起队内报名"
- 设置信息：
  - 报名上限人数（例如 15 人）
  - 报名截止时间（默认：开赛前 6 小时，可调整）
- 发起后，推送订阅消息至全体队员

#### 3.4.2 球员报名

- 球员在小程序内看到报名入口
- 点击"我要参加"确认报名
- 超过上限人数后，自动进入候补队列
- 候补球员在有人退出时按顺序递补，并收到通知

#### 3.4.3 取消报名

- **截止时间前取消：** 正常退出，候补自动递补
- **截止时间后取消：** 允许取消，但记录一次"临时退出"（影响个人出勤统计）
- V1 不做金钱惩罚，仅做数据记录

#### 3.4.4 报名状态看板（队长视角）

- 已报名人数 / 上限
- 候补人数
- 每位球员的报名状态（已报名 / 候补 / 已取消）
- 队长可手动调整名单（添加未注册的临时外援，记为"访客"）

**产品决策说明：**
- **V1 不做"报名预扣费"**：调研中用户提出报名时冻结 50% 费用来约束鸽子行为。但微信支付不支持预授权冻结（需企业资质+专项能力申请），V1 用出勤记录 + 信用分来软约束，V2 根据运营数据评估是否需要接入预付能力。

### 3.5 模块五：费用结算

这是队长痛点最深的模块（"讨债鬼"问题）。

#### 3.5.1 创建赛后账单（队长操作）

比赛结束后，队长进入比赛详情页 → "生成账单"：

| 步骤 | 操作 |
|------|------|
| 1. 确认费用项 | 场地费（自动带入约战时填写的金额）+ 手动添加其他费用（水费、裁判费等） |
| 2. 确认实到人员 | 从报名列表中勾选实际到场的人（未到场的取消勾选） |
| 3. 设置特殊角色 | 可标记"门将免单"、"半价"等特殊费率（V1 支持免单和自定义金额两种） |
| 4. 系统计算 | 自动计算每人应付金额（精确到分） |
| 5. 发布账单 | 每位队员的小程序内收到账单推送 |

#### 3.5.2 球员支付

- 球员打开账单 → 查看明细（总费用、分摊人数、个人金额）→ 点击"微信支付"
- **支付方式：** 微信支付直接转账至队长的微信商户号（或个人收款码）
- **V1 实现方案：** 生成收款二维码 / 跳转微信支付（因微信支付商户号申请门槛较高，V1 可先支持"标记已付"手动确认方案作为兜底）

#### 3.5.3 收款状态追踪

队长可见每位球员的支付状态：

- ✅ 已支付
- ⏳ 待支付
- ❌ 逾期未支付（超过 48 小时）

**逾期提醒规则：**
- 账单发布后 24 小时，系统通过订阅消息提醒未付款球员（需球员提前授权）
- 48 小时后仍未支付，队长端标红显示

**产品决策说明：**
- **不做"球队钱包/预充值"**：调研中用户最想要"自动扣费"功能。但在中国，小程序运营方持有用户预存资金属于"资金池"行为，需持有《支付业务许可证》，监管风险极高。V1 采用"赛后即时支付"模式，以系统催收替代人工催款，解决核心尴尬。未来可考虑与持牌支付机构合作实现预付卡能力。
- **"系统催"替代"队长催"**：这是调研中用户的原话——"系统催那是规则，我催那是不信任"。即使 V1 只能做到"标记 + 提醒"，也已经解决了核心的社交尴尬问题。

### 3.6 模块六：赛后记录

#### 3.6.1 比分确认（双向握手）

比赛结束后，需双方队长确认比分：

1. **主队队长提交比分**（例如：暴风 5 : 3 咸鱼）
2. **客队队长确认**：
   - 情况 A：确认一致 → 比分生效，进入数据录入阶段
   - 情况 B：有异议 → 客队提交己方版本，系统提示"比分不一致，请双方队长私下核实后重新提交"
   - 情况 C：客队 48 小时不操作 → 以主队提交为准（防止拖延）
3. **双方均未提交**（72 小时内）→ 标记为"未记录比赛"，不计入战绩

**产品决策说明：**
- **不做复杂仲裁系统**：调研中用户提出了基于证据的仲裁、裁判终裁等方案。但仲裁系统需要大量人工运营成本，V1 采用"双向确认 + 超时默认"的轻量方案。对于恶意不确认的情况，通过信用分惩罚机制处理。
- **"关门自治"原则**：对手不关心你队谁进的球，只关心总比分。因此比分确认是跨队行为，数据分配是队内行为，两者解耦。

#### 3.6.2 队内数据录入（队长操作）

比分确认后，队长为己方填写详细数据：

- **进球分配：** 系统根据己方进球数生成对应数量的"空槽"，队长将进球分配给具体球员
  - 支持"未知/外援"选项（临时参赛的未注册人员）
- **助攻分配：** 同进球分配逻辑（可选填）
- **MVP 投票发起：** 队长确认数据后，自动触发队内 MVP 投票

#### 3.6.3 MVP 投票

- **投票资格：** 本场报名且实际到场的队员
- **投票规则：** 每人投 1 票，不可投自己
- **投票窗口：** 赛后 24 小时内
- **结果展示：** 得票最多者获得本场 MVP 标识（平票则并列）

#### 3.6.4 战报生成（V2）

> V1 仅做数据录入和展示，战报海报生成功能列入 V2。

V2 预期能力：
- 自动生成比赛战报图片（含双方队名、比分、进球名单、MVP）
- 支持一键分享到微信群/朋友圈
- 提供 2-3 种海报模板可选

### 3.7 模块七：信用与评价体系

#### 3.7.1 信用分

每支球队拥有一个公开可见的信用分，反映其履约可靠程度。

| 行为 | 信用分变动 |
|------|-----------|
| 正常完赛（双方确认比分） | +2 |
| 赛前 24 小时以上取消 | 0（不扣分） |
| 赛前 6-24 小时取消 | -5 |
| 赛前 6 小时内取消 | -15 |
| 无故不到场（对方到场签到，己方未签到） | -30 |
| 连续 3 场正常完赛 | 额外 +5（连续履约奖励） |

**初始信用分：** 100 分
**显示规则：**
- ≥ 90：绿色（信誉良好）
- 70-89：黄色（一般）
- < 70：红色（需警惕），约战列表中强制置底显示

**产品决策说明：**
- **不做"连坐机制"和"封号"**：调研中用户提出信用分低的队长不可创建新队等惩罚。V1 阶段不做如此激进的惩罚（用户基数小时，过度惩罚会导致用户流失），仅通过展示层面的"降权"让市场自然筛选。
- **签到机制简化**：调研中建议用 GPS + 照片验证签到。V1 仅做"队长手动确认到场"（主队确认客队到场），不做 GPS 强制校验（定位精度不够且用户体验打断感强）。

#### 3.7.2 赛后互评（V1 简化版）

比分确认后，双方队长可对对方球队进行评价：

- **评价形式：** 标签选择（非自由文本，避免辱骂）
- **正面标签：** 准时到场 / 球风干净 / 实力匹配 / 氛围友好 / 费用爽快
- **负面标签：** 迟到 / 动作粗野 / 实力不符 / 垃圾话多 / 费用拖延
- **展示规则：**
  - 单次评价不公开展示
  - 同一标签被 **3 个以上不同对手** 标记后，在球队主页聚合展示
  - 这避免了"一次冲突定性"的问题，只有长期行为模式才会形成公开标签

**产品决策说明：**
- **V1 不做"双盲互评"**：调研中用户建议互评在双方都提交后才可见对方评价。此功能的前端体验和后端逻辑较复杂，V1 先做单向评价（我评价对方），V2 引入双盲机制。
- **不做球风/实力的强制修正**：调研中用户希望"连续 3 个对手说你强，系统强制升级"。V1 不做强制调整（自主定位优先），但在球队主页展示"实力超出描述"标签作为提示。

球队主页展示以下统计数据（基于历史比赛自动汇总）：

- 总比赛场次 / 胜 / 平 / 负 / 胜率
- 近 10 场战绩趋势
- 总进球数 / 总失球数

#### 3.8.2 球员个人档案

每位球员拥有个人数据页：

- 所属球队
- 擅长位置
- 累计出场次数
- 累计进球 / 助攻
- 获得 MVP 次数
- 出勤率（报名场次 / 总比赛场次）

> **社交货币价值：** 调研表明，球员个人档案是驱动普通球员使用小程序的核心动力——"为了保住射手榜第一的位置，刮风下雨他都会来"。

### 3.9 模块九：消息与通知

微信小程序的订阅消息机制决定了通知策略：

| 场景 | 通知方式 | 触发条件 |
|------|----------|----------|
| 有人应战 | 订阅消息 | 队长发布约战后，有对手应战 |

### 3.10 模块十：酒水服务（"庆功时刻"）

调研中用户对"球队定制酒"表达了极强的付费意愿，且赛后聚餐饮酒是业余足球文化的刚性场景（用户原话："踢球是借口，喝酒吹牛才是正事"）。本模块是产品的核心商业化切入点之一。

#### 3.10.1 合规架构：场景导流模式

**核心原则：Spot On 不卖酒，只"种草"和"带路"。**

在中国，线上销售酒类需持有**食品经营许可证（含酒类经营范围）**。若 Spot On 平台自身作为卖家，需要承担进货、仓储、物流、售后等全链路责任，且需办理相关资质，这与轻量工具型小程序的定位冲突。

**规避方案——"场景导流 + 外部成交"：**

```
Spot On 小程序（场景入口）          经销商小程序/店铺（持证卖家）
        │                                    │
        ├─ 提供赛后"庆功时刻"场景             │
        ├─ 展示酒水商品信息（图片、文案、价格） │
        ├─ 用户点击"去购买"                   │
        │         ──── 跳转 ────►              │
        │                           ├─ 完成下单、支付、发货
        │                           ├─ 持有食品经营许可证
        │                           └─ 承担售后责任
        │
        └─ Spot On 角色 = 广告展示 + 流量导流
           不碰商品、不碰资金、不碰物流
```

**法律定性：**
- Spot On = **广告信息服务方**（展示商品信息、引导跳转），类似"美食推荐"而非"外卖平台"
- 经销商 = **实际销售方**，持有食品经营许可证，独立承担商品质量和售后责任
- 收入模式 = 经销商向 Spot On 支付**广告费/导流佣金**（CPS 按成交抽佣或 CPM 按展示计费），而非 Spot On 赚取商品差价
- 微信生态支持 = 小程序间跳转（`wx.navigateToMiniProgram`）是微信官方能力，合规无障碍

#### 3.10.2 产品功能设计

##### 场景一：赛后"庆功时刻"推荐（V2）

**触发时机：** 比分确认完成后，比赛详情页底部出现"庆功时刻"入口卡片。

**页面内容：**
- "恭喜暴风联队获胜！来瓶好的庆祝一下？" / "虽败犹荣，兄弟们辛苦了，犒劳一下自己"
- 推荐 2-3 款酒水商品（啤酒为主、白酒为辅），带图片+价格+一句话文案
- 点击商品 → **跳转至经销商的微信小程序商城**完成下单

**推荐策略：**
- 赢球推荐："冠军畅饮装"（啤酒组合）、"庆功特供"（白酒）
- 输球推荐："东山再起"（啤酒安慰装，文案走温情路线）
- 高温天推荐冰啤，冬季推荐白酒
- 推荐商品由经销商后台配置，Spot On 仅做展示

##### 场景二：赢球彩头（V2）

**玩法：** 约战时，双方队长可勾选"酒水彩头"。

**流程：**
1. 约战发布时，主队勾选"本场设置酒水彩头"
2. 选择彩头商品（从经销商提供的商品池中选择，例如"一箱定制啤酒 ¥128"）
3. 客队接受应战时确认彩头
4. 比赛结束，**输方队长收到推送**："愿赌服输！点击为对手送上一箱庆功酒"
5. 点击后跳转经销商小程序下单，**收货地址填写赢方队长地址**（或赛后聚餐餐厅地址）

**关键设计：**
- 彩头是"社交承诺"而非"强制支付"——Spot On 不代收代付，不冻结资金
- 不履行彩头不会有金钱惩罚，但会在对战记录中标记"彩头未兑现"（社交压力 > 金钱惩罚）
- 赢方也可主动放弃彩头（体现风度）

##### 场景三：球队定制酒（V2-V3）

**需求来源：** 用户原话——"如果能做球队定制罐身的啤酒，印上暴风联队 Logo，一箱起订，我会买！"

**流程：**
1. 队长进入"球队商城" → "定制庆功酒"
2. 选择酒品基底（经销商提供的可定制 SKU）
3. 上传球队 Logo / 选择文案模板（如"暴风联队 · 队长之选"、"年度射手王纪念版"）
4. **Spot On 提供定制设计工具**（简单的贴图/文字编辑器，生成设计稿预览）
5. 确认设计 → **跳转经销商小程序下单**（经销商负责生产、包装、发货）

**Spot On 的角色：**
- 提供设计工具（前端模板编辑器）—— 这是纯信息服务，不涉及酒类许可
- 将设计稿传给经销商系统 —— 技术对接
- 不参与生产、不持有库存、不处理退换货

##### 场景四：赛事赞助酒水（V3）

**面向赛事/杯赛组织场景：**
- 经销商以"赞助商"身份为 Spot On 上的杯赛提供奖品酒水
- 赛事页面展示赞助商品牌露出
- 获奖球队通过经销商渠道领取奖品

**本质是 B2B 广告赞助**，与酒类零售许可完全无关。

#### 3.10.3 经销商后台（独立于 Spot On 主小程序）

经销商需要一个管理后台来配置商品和追踪数据：

| 功能 | 说明 |
|------|------|
| 商品管理 | 上架/下架推荐商品，设置图片、价格、文案、跳转链接 |
| 定制模板管理 | 管理可定制的 SKU 和设计模板 |
| 数据看板 | 查看导流点击量、跳转转化率、成交额（需经销商侧回传数据） |
| 场景配置 | 配置不同场景（赢球/输球/赛事）下推荐的商品组合 |

> 此后台可作为 Spot On 管理后台的一部分（Web 端），不涉及小程序端。

#### 3.10.4 合规要点清单

| 事项 | 要求 | Spot On 责任 |
|------|------|-------------|
| 食品经营许可证（酒类） | 实际销售方必须持有 | 经销商持有，Spot On 不需要 |
| 广告法合规 | 酒类广告不得含有"饮酒有益健康"等违规内容 | Spot On 负责审核展示文案 |
| 未成年人保护 | 不得向未成年人销售酒类 | 经销商小程序负责校验，Spot On 可在跳转前提示"未成年人请勿购买" |
| 小程序跳转规则 | 需双方小程序关联同一公众号，或同一开放平台账号 | 提前做好小程序关联配置 |
| 资金流向 | 所有支付发生在经销商小程序内 | Spot On 不碰资金流水 |
| 佣金结算 | 经销商按月向 Spot On 支付导流佣金 | 双方签订广告服务合同，开具广告服务发票 |

## 七、商业模式（分阶段）

### 7.1 V1 阶段：免费工具，积累用户

- 所有核心功能免费
- 不收取任何交易手续费
- 目标：验证产品价值、积累种子用户（目标 500 支球队）

### 7.2 V2 阶段：增值服务试水

| 收入来源 | 模式 | 预期 |
|----------|------|------|
| 战报海报高级模板 | 免费模板 + 付费精美模板（3-5 元/张） | 低频，品牌建设 |
| 球队数据高级统计 | 基础免费，详细赛季报告付费（队长订阅 19.9 元/月） | 中频 |
| 裁判预约服务 | 平台抽佣 10-15% | 中频 |
| **酒水导流（"庆功时刻"）** | **经销商支付导流佣金（CPS 按成交 10-15% 抽佣）** | **高频场景，核心商业化入口** |
| **赢球彩头酒水** | **同上，叠加社交传播带来的转化溢价** | **中频，客单价高** |

### 7.3 V3+ 阶段：平台生态

| 收入来源 | 模式 |
|----------|------|
| 摄影/摄像预约 | 平台抽佣 |
| 单场意外险嵌入 | 与保险公司合作，按单抽佣 |
| **球队定制酒** | **经销商生产+发货，Spot On 提供设计工具+导流，收取设计服务费+佣金** |
| **赛事赞助酒水** | **经销商以赞助商身份提供奖品，支付品牌露出费** |
| 装备/定制周边 | 同导流模式，与供应商合作 |
| 赛事/杯赛组织 | 报名费抽佣 + 赞助商广告 |

**产品决策说明：**
- **V1 不做任何商业化**：调研中用户表达了多种付费意愿，但产品初期最重要的是验证核心价值和积累用户密度。过早收费会拉高获客门槛。
- **酒水服务采用"场景导流"模式而非自营电商**：Spot On 定位为场景入口和广告服务方，不碰商品、不碰资金、不碰物流，无需食品经营许可证。经销商以独立小程序完成实际销售，Spot On 收取导流佣金。详见功能模块 3.10。
- **保险嵌入需牌照合规**：互联网保险销售需持有或合作保险经纪牌照，V3 阶段引入时需提前做合规准备。

## 八、运营策略

### 8.1 冷启动策略

业余足球的组织关系以"队长-球员"为核心，队长是关键节点：

1. **队长驱动：** 重点运营队长群体（种子用户），由队长在微信群内推广小程序给队员
2. **队长特权：** 提供免费的"队长数据大屏"（赛季汇总），增强队长使用动力
3. **病毒传播：** 约战本身就是跨队传播——A 队约 B 队，B 队如果没注册就需要引导注册
4. **战报分享：** 赛后战报是天然的传播物料，分享到朋友圈/微信群带来自然流量

### 8.2 留存策略

- **数据绑定：** 球员的个人档案（进球、出场）形成数据资产，迁移成本高
- **社交绑定：** 球队关系链（队友、对手）沉淀在平台内
- **习惯养成：** 每周固定的报名-踢球-结算流程形成使用惯性
- **榜单激励：** 射手榜、出勤榜、MVP 榜等排名激发竞争心理

## 九、风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| 信用分被恶意刷分 | 评价体系失真 | 标签聚合阈值（3 次以上才展示）+ 异常检测 |
| 队长不愿推广给队员 | 产品核心循环断裂 | 降低球员使用门槛（微信一键登录、无需注册）；给队长提供足够的管理便利 |
| 支付通道合规风险 | 资金安全 | V1 不持有用户资金；每笔交易直接走微信支付；不做钱包预充值 |
| 约战双方恶意行为 | 用户体验受损 | 信用分惩罚 + 标签系统 + 举报通道 |
| 微信订阅消息授权率低 | 关键通知触达不到 | 在核心操作节点引导授权；辅以微信群内小程序卡片分享 |
| 内容安全风险 | 小程序被封禁 | 所有 UGC 内容（球队名、简介、头像）经微信内容安全 API 审核 |
